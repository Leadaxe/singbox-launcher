# SPEC 045 — STATE_CONFIG_DECOUPLING

**Тип:** F (Feature) · **Статус:** N (New, в планировании) · **Релиз:** следующий major (v0.9 / v1.0).

## Контекст и проблема

Сейчас «Wizard» в лаунчере — монолит: при Save он одновременно **(а)** пишет `state.json` (декларативное состояние: vars, rules, sources, DNS settings) и **(б)** мутирует `config.json` (исполняемый шаблон для sing-box с подставленными outbounds и plaсeholder'ами, развернутыми из vars). Побочные эффекты этой связки:

1. **Семантически размазанный dirty-marker** (SPEC 043). Маркер `*` на Update загорается при любом Save визарда, но:
   - `config.json` переписывается **всегда** при закрытии визарда (шаблон + уже лежавшие в памяти outbounds-ноды);
   - подписки при этом **не рефетчатся** — новые URL / добавленные источники / изменённый skip ничего не скачивают. Пользователь вручную давит Update или ждёт auto-update.
   - При этом маркер `*` семантически корректен только для правок `ParserConfig.Proxies` (URL / skip / tag_prefix). Для правок `tun_stack`, DNS, rules, log_level нужен **Restart sing-box**, а не Update (= перезапуск парсера).
2. **Нет понятия «draft state»** — ты не можешь поправить настройку, закрыть визард и потом отрендерить разные варианты `config.json` из одного state. State и сгенерированный config склеены в обе стороны.
3. **Невозможно разделить «что поменялось в state» и «что уже попало в работающий config»** — нет промежуточной сущности, которая могла бы сказать «state изменён, но build пока не перекатывал».
4. **Отсутствие реактивной модели** — весь UI пересобирается через один `UpdateConfigStatusFunc`, который работает как coarse-grained broadcast «что-то где-то изменилось, обновите всё». Это тянет за собой чрезмерные IO-чтения и шум в логах. Формально это отдельный TODO, но по факту он вырос из той же корневой архитектурной проблемы — нет типизированных сущностей, между которыми можно было бы объявить явные события.

## Источник вдохновения

**LxBox (мобильный клиент)** — уже построен на модели «state отдельно, build отдельно». См. `/Users/macbook/projects/LxBox/` для канонической реализации. В ночном ревью паритета с LxBox (22.04.2026) все пункты mobile → desktop подтягивания сошлись естественно, **кроме** этого — решили, что это работа на следующий серьёзный релиз, потому что она требует миграции формата state и инвалидации кэша outbounds.

## Желаемое поведение

### 1. Две раздельные операции

| Операция        | Пишет             | Читает                                   | Триггер                                          |
|-----------------|-------------------|------------------------------------------|--------------------------------------------------|
| **Wizard Save** | `state.json`      | текущий `state.json`                     | кнопка Save в визарде                            |
| **Build Config**| `config.json`     | `state.json` + subscription cache        | кнопка Update / cron auto-update / explicit call |

Wizard Save **не** трогает `config.json` и **не** триггерит sing-box reload. Build Config — единственная точка, генерирующая `config.json`, и именно она даёт команду на reload.

### 2. Два сигнала в UI

| Сигнал                   | Условие                                                                                                             |
|--------------------------|---------------------------------------------------------------------------------------------------------------------|
| `*` на кнопке **Update** | `state.json` имеет изменения в `Proxies` / списке источников / skip / tag_prefix, которые не попали в последний Build |
| маркер на **Restart**    | шаблон (tun / dns / rules / log_level / etc.) менялся, но sing-box продолжает работать со старым `config.json`      |

Два маркера существуют одновременно и независимо. Пользователь видит, что именно «протухло»: только подписки — жмёт Update; шаблон — жмёт Restart; оба — одно за другим.

### 3. Новое наименование

Термин **Wizard** описывает сейчас и «редактор state», и (косвенно) «инициатор генерации config». После разделения Wizard → **редактор state** (без build-стороны). Кандидаты на новое имя — подобрать в PLAN после деёп-дайва LxBox (у мобилки уже есть устоявшаяся терминология, её стоит перенять). Рабочий placeholder: **Configurator** / **State Editor**.

### 4. Реактивная модель (связанный, но отдельный TODO)

После того как state и config разделены — естественно ввести **типизированные события**:
`StateChanged(kind)`, `ConfigBuilt`, `VpnStateChanged`, `SubscriptionUpdated`, `ProxyActiveChanged`, …

Виджеты подписываются точечно на то, что их касается, вместо broadcast-«обновите всё». Это **вторая SPEC**, логически растущая из 045 и запущенная после неё (или параллельно в конце 045 как миграционный шаг).

## Критерии приёмки

1. Save в визарде не мутирует `config.json`. Проверка: `diff` `config.json` до и после Save без промежуточного Build должен быть пустой.
2. Два отдельных UI-маркера: `*` на Update и отдельный маркер на Restart (экспонированный через StateService). Юнит-тест, дёргающий «изменил только source» → горит только Update; «изменил только tun_stack» → горит только Restart; «изменил оба» → горят оба.
3. Build Config — единственный код-путь, вызывающий запись `config.json`. Grep по `os.WriteFile.*config\.json` / `atomicwrite.*config\.json` в проекте даёт один call-site после миграции.
4. Миграция со старого формата: существующие `state.json` / `config.json` / `settings.json` читаются без потери данных; версия state-файла инкрементится (см. существующую политику `state.json v4` из SPEC 002).
5. SPEC на event-driven модель (045-дочерний или 046) заведён и связан.
6. Вся терминология UI последовательна (если решим переименовать Wizard → Configurator, переименовано везде: вкладки, i18n-ключи, docs).

## Риски

1. **Миграция state** — старые установки имеют `state.json` + `config.json` в старой связке. Нужно чётко отработать «первый запуск после апдейта»: подтянуть state, лениво пересобрать config при первой проверке (или при первом Update).
2. **Кэш outbounds** — сейчас outbounds, полученные при последнем парсинге подписок, лежат в памяти (через `PopulateParserMarkers`) и пишутся в `config.json`. В новой модели их надо хранить **отдельно от config** (например, как `outbounds.cache.json` или в памяти `StateService`) — иначе Build Config без вызванного парсера не сможет собрать рабочий конфиг.
3. **UX-шок** — если переименовать «Wizard» на ровном месте без явной причины в UI, пользователь растеряется. Нужна понятная формулировка в release-notes и, возможно, tooltip на вкладке «раньше называлось Wizard» (один релиз, потом убираем).
4. **Большая площадь изменений** — затрагивает `ui/wizard/*`, `core/config/*`, `state/*`, `main.go` bootstrap, все callsites `UpdateConfigStatusFunc`. Стоит бить на несколько PR'ов по слоям.
