# SPEC: расширенный редактор источника (первая вкладка визарда) и исключение нод из глобальных селекторов

**Статус:** N (новая). **Тип:** F (feature).

---

## Проблема

1. **Первая вкладка (Sources):** настройки источника размазаны по списку (префикс в строке) и отдельному окну **View** только со списком серверов. Нет одного места для логики «локальные auto/select» и связанных полей `ParserConfig`.
2. **Парсер:** ноды из всех `proxies[]` всегда попадают в выборку для **глобальных** `ParserConfig.outbounds` (`filterNodesForSelector(allNodes, …)` в `buildOutboundsInfo`). Нельзя оставить ноды только за локальными `urltest`/`selector`, чтобы глобальные группы ссылались на теги локальных auto/select, а не на десятки отдельных тегов серверов.

## Цель

1. Заменить кнопку **View** на **Edit** и открывать окно с **двумя внутренними вкладками**:
   - **Настройки** — префикс (и при необходимости постфикс/маска/skip, если уже редактируются только в JSON; минимум — префикс по требованию), галочки создания локальных outbounds **auto** (`urltest`) и **select** (`selector`); для **select** — в список `outbounds` селектора входит локальный **auto**, и **`default`** селектора указывает на этот **auto**.
   - **Просмотр** — текущее поведение окна View: список распарсенных серверов источника (кэш превью / загрузка как сейчас).

2. Расширить модель **`proxies[]`** в `ParserConfig`: флаг **«исключить из глобальных списков»** — ноды этого источника **не участвуют** в подборе `filteredNodes` для **глобальных** селекторов; доступ к ним для глобальных групп — через теги локальных **auto**/**select** (добавляемых галочками и/или уже заданных в `proxies[].outbounds`).

3. Документировать формат в **`docs/ParserConfig.md`** и при необходимости **`bin/wizard_template.json`** (без ломки существующих конфигов).

## Требования к UI

| Элемент | Поведение |
|---------|-----------|
| Кнопка в строке источника | Подпись **Edit** (локали EN/RU), открывает то же окно, что раньше View, с двумя подвкладками. |
| Вкладка «Настройки» | Поле **префикс тегов** (`tag_prefix`), синхронизация с моделью и `ParserConfigJSON` как у текущего поля в списке (или перенос поля из строки списка в окно — см. PLAN; дублирования вводить не нужно). |
| Чекбокс «локальный auto» | Создаёт/удаляет в `proxies[i].outbounds` запись типа **`urltest`** с согласованным **тегом** (стабильное именование — в PLAN; не конфликтовать с глобальными `auto-proxy-out` без явного правила). |
| Чекбокс «локальный select» | Создаёт/удаляет запись типа **`selector`**; если включён **auto**, в `outbounds` селектора входит тег локального **auto**, **`options.default`** (или поле `default` согласно текущему генератору) = этот тег; в список также входят все ноды источника (как при обычном селекторе «все сервера источника» — уточнение в PLAN через `filters` или явный список). |

## Требования к парсеру и модели

1. Новое поле в **`ProxySource`** (`core/config/configtypes`, алиас в `core/config`): boolean, по умолчанию `false`. При `true`: при расчёте **глобальных** селекторов в `buildOutboundsInfo` ноды, загруженные из этого индекса `proxies`, **исключаются** из `allNodes` для целей `filterNodesForSelector` **только для глобальных** `OutboundConfig` (локальные `proxies[i].outbounds` по-прежнему используют `nodesBySource[i]`).

2. Для сопоставления «нода ↔ индекс источника» потребуется явная привязка на этапе генерации (например поле **`SourceIndex`** у `ParsedNode` или параллельный срез индексов), задаваемая в цикле `GenerateOutboundsFromParserConfig` при вызове `loadNodesFunc`. Обратная совместимость: для кода без установки индекса — считать ноду глобальной (не исключать), либо всегда выставлять индекс в генераторе.

3. **Версия `ParserConfig`:** по возможности оставить **4** с опциональными полями; при необходимости ввести **5** с правилом миграции в `core/config/parser/migrator.go` и обновлением `ParserConfigVersion` / документации.

4. Локальные **auto/select**, создаваемые из UI, сериализуются в `proxies[].outbounds` в том же формате, что и ручные записи (типы `urltest` / `selector`, `filters` / `addOutbounds` / `options` — согласовать с `GenerateSelectorWithFilteredAddOutbounds` и существующими примерами в **ParserConfig.md**).

## Критерии приёмки

- [ ] В строке источника кнопка **Edit**; в окне две подвкладки: настройки (префикс, три чекбокса по смыслу выше) и просмотр серверов.
- [ ] Включение «локальный select» при включённом «локальный auto» даёт селектор с `default` на auto и `outbounds`, содержащим тег auto и ноды источника.
- [ ] Включение «исключить из глобальные списки» меняет результат генерации: глобальные селекторы не получают в свой список ноды этого источника по фильтрам; локальные селекторы источника не ломаются.
- [ ] Старые `config.json` без новых полей парсятся как раньше.
- [ ] `docs/ParserConfig.md` описывает новые поля и сценарий «только через локальные auto/select в глобальные addOutbounds».
- [ ] `go vet ./...`, `go build ./...`, `go test ./...` (GUI-пакеты — по CONSTITUTION).

## Зависимости и риски

- Глобальные селекторы должны явно включать теги локальных **auto**/**select** (через `addOutbounds` или фильтры по тегу), иначе трафик через глобальный `proxy-out` не увидит исключённые ноды — это поведение конфигурации; SPEC не обязывает автоматически переписывать глобальный `outbounds`, но в PLAN можно заложить опциональную подсказку в UI.
- Конфликты тегов локальных и глобальных селекторов — существующее предупреждение в логах; именование локальных тегов из визарда должно минимизировать коллизии (например префикс по `tag_prefix` или индекс источника).

## Вне объёма (явно)

- Автоматическое добавление тегов локальных групп во **все** глобальные селекторы без действия пользователя.
- Изменение формата подписок или загрузчика нод, кроме пометки индекса источника.
