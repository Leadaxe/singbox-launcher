# 083-B-W — RUTRACKER `SSL_ERROR_RX_RECORD_TOO_LONG` через WARP-конфиги

**Тип:** Bug (диагностика) · **Статус:** W (Wait — решение за владельцем; код не менялся)
**Дата диагностики:** 2026-07-12 · **Платформа репорта:** desktop (macOS/Windows)

---

## 1. Жалоба (as reported)

> С сегодняшнего дня на десктопе singbox launcher перестал открывать `https://rutracker.org`
> через **любой конфиг amnezia warp** с ошибкой `SSL_ERROR_RX_RECORD_TOO_LONG`.
> Amnezia-клиент эти же конфиги открывает.

## 2. Итог диагностики (TL;DR)

**Это НЕ дефект туннеля и почти наверняка НЕ дефект кода лаунчера.** Это следствие
**split-tunnel маршрутизации**: `rutracker.org` уходит **мимо VPN (direct)**, а российский
DPI на запрос к `rutracker.org:443` инжектит plaintext HTTP-ответ (страницу-заглушку РКН)
вместо TLS-записи. Firefox видит первый байт ответа `0x48` (`H` от `HTTP/…`) вместо `0x16`
(TLS handshake record) → `SSL_ERROR_RX_RECORD_TOO_LONG`. Amnezia открывает те же конфиги,
потому что гонит **весь** трафик через WARP (full-tunnel), а лаунчер по шаблону раскидывает
трафик правилами и часть доменов пускает напрямую.

Оценка вероятности, что «фикс одной строкой в коде» устранит проблему: **низкая**.
Автономный код-фикс **не вносился** — он рискует сломать существующие сценарии
маршрутизации у других пользователей. Ниже — доказательная база и варианты решения
(это UX/шаблонное, не bug-fix решение).

## 3. Что проверено (воспроизведение)

Тесты выполнены отдельным экземпляром `sing-box` (по captured PID, на портах 18080;
живой TUN лаунчера не трогался), ядро `1.14.0-lx.1-rc.17`.

| # | Сценарий | Результат |
|---|----------|-----------|
| 1 | Регистрация WARP-аккаунта через Cloudflare API (`POST /v0a4005/reg`) | ✅ HTTP 200, получены peer pubkey / addresses / client_id |
| 2 | `curl -x socks5h://…:18080 -I https://rutracker.org` через **plain-WG WARP** (mtu 1280, reserved из client_id) | ✅ `HTTP/2 301`, `server: cloudflare` — открывается |
| 3 | `curl -I https://rutracker.org` через **живой TUN лаунчера** (route-правила рабочего config.json) | ✅ TLS 1.3 handshake OK, `HTTP/2 301` — открывается |
| 4 | DNS-резолв `rutracker.org` (system → hijack-dns; DoH через туннель) | ✅ `104.21.32.39` / `172.67.182.196` (Cloudflare), одинаково по всем путям |

Симптом `SSL_ERROR_RX_RECORD_TOO_LONG` **на текущем ядре/шаблоне не воспроизведён** —
это подтверждает, что дефект не в способе, которым лаунчер строит WARP-endpoint.

## 4. Доказательство корневой причины

`rutracker.org` присутствует как **`domain_suffix`** в списке заблокированных РФ-доменов,
который грузят пресеты шаблона:

```
geosite-ru-blocked.srs      → domain_suffix: "rutracker.org"   (10 хостов rutracker.*)
geosite-ru-blocked-all.srs  → domain_suffix: "rutracker.org"   (115 хостов rutracker.*)
```
(источник: `runetfreedom/russia-v2ray-rules-dat`, ветка `release`,
`sing-box/rule-set-geosite/`; используется пресетами `ru-blocked`, `block-ads`, `russian`.)

Механизм `SSL_ERROR_RX_RECORD_TOO_LONG` = ответ **не в TLS-формате** на порт 443. Для
заблокированного в РФ ресурса это классическая **DPI-инъекция заглушки** провайдером,
срабатывающая только когда соединение идёт **напрямую**, а не через VPN.

**Почему трафик может идти direct именно в лаунчере (а не в Amnezia):**
- пресет `ru-blocked` (который завернул бы заблокированные домены в VPN) **не включён**
  у пользователя, и `rutracker.org` попадает под `route.final`, а `final` указывает на
  direct-подобный outbound; **или**
- активен пресет `russian`/`ru-inside` с `@out = direct` для «российских» доменов, а
  свежая версия rule-set затянула rutracker-домены в этот набор («с сегодняшнего дня» =
  обновился remote rule_set по расписанию `parser.reload`); **или**
- WARP-endpoint с AmneziaWG-обфускацией у пользователя фактически не поднимается
  (см. историю чата: «напрямик warp из программы блокируется, через detour работает»;
  завышенный MTU у AWG → handshake OK, данных нет), и трафик деградирует на direct-путь.

Все три ветки сводятся к одному: **rutracker уходит в direct → DPI-заглушка → ошибка TLS**.

## 5. Контекст сообщества (Telegram)

- Слово «rutracker» в чате проекта **не встречается ни разу**.
- `SSL_ERROR_RX_RECORD_TOO_LONG` встречался **один раз** (22.01.2026) — на бесплатной ноде
  с кривым TLS (joyreactor.cc), диагноз сообщества: «кривая нода», не дефект лаунчера.
- WARP через встроенный генератор жаловались, что «блокируется напрямую, работает через
  detour»; автор обещал «докрутить встроенный WARP-генератор до AWG 1.5» (15.06.2026).

## 6. Рекомендации (решение — UX/шаблонное, не код-фикс)

Ранжировано по «пользе/риску»:

1. **Диагностический хинт в UI.** Если активный конфиг ловит TLS-подобную ошибку на
   заблокированном домене — подсказать: «домен заблокирован в РФ и идёт напрямую; включите
   пресет "Заблокированные РФ → VPN" или задайте `route.final = <ваш VPN-канал>`». Дёшево,
   безопасно, закрывает класс жалоб (перекликается с FAQ #6 «почему пропал интернет при
   Final=direct» и #8 «куда смотреть при ошибке ядра»).
2. **Шаблон: по умолчанию заворачивать `ru-blocked` в VPN** (не в direct) при первом
   запуске wizard — чтобы заблокированные РФ-домены не утекали в direct-заглушку.
3. **WARP-генератор до AWG 1.5/2.0 + full-tunnel по умолчанию** — закрывает ветку «WARP не
   поднимается напрямую» (реализуется в [084-F-N-WARP_GENERATOR](../084-F-N-WARP_GENERATOR/SPEC.md)).
4. **Kill-switch/route.final-guard**: предупреждать, когда `final` = direct-подобный
   outbound (риск утечки заблокированного трафика).
