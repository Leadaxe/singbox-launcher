# Спецификация: Вкладка Raw в диалоге Add/Edit Rule

## 1. Проблема

### 1.1 Общее с Edit Outbound

При добавлении/редактировании **Outbound** в визарде есть вкладка **Raw**: многострочное поле с JSON конфига outbound'а, можно писать всё на лету. При добавлении/редактировании **Rule** такого режима нет — есть только тип «Custom JSON», который по сути даёт одно многострочное поле в общем виде формы. Нет отдельной вкладки «Raw», как в Edit Outbound.

### 1.2 Цель

Добавить в диалог **Add Rule / Edit Rule** вторую вкладку **Raw** по аналогии с Edit Outbound:

- **Вкладка 1 (Form):** текущий интерфейс — Rule name, тип правила (чекбоксы), поля по типу (IP, Domains, Processes с Match by path, Custom JSON), Outbound.
- **Вкладка 2 (Raw):** многострочное поле с JSON **объекта правила** (route rule): те же поля, что попадают в `route.rules[]` (domain, ip_cidr, process_name, process_path_regex, rule_set, outbound, action и т.д.). Пользователь может писать правило целиком на лету; при сохранении используется этот JSON (с валидацией).

Так сохраняется единообразие с Edit Outbound и даётся удобный способ задать произвольное правило. Любое изменение пользователя в Raw по сути даёт «кастомное» правило: нет гарантий, что содержимое влезает в шаблон формы (IP/Domain/Process), и не требуется соответствие ему — при сохранении используется JSON как есть (с проверкой только на валидность и наличие outbound/action).

## 2. Требования

### 2.1 Вкладки в диалоге Add Rule / Edit Rule

- В диалоге две вкладки: **Form** и **Raw** (порядок: сначала Form, затем Raw).
- Переключение вкладок не сохраняет правило само по себе; при **Save** сохраняется:
  - если активна **Form** — правило, собранное из формы (как сейчас);
  - если активна **Raw** — правило, распарсенное из JSON на вкладке Raw (объект route rule; должен содержать как минимум `outbound` или `action`; при необходимости подставлять outbound из выбора в форме — уточнить в реализации).

### 2.2 Вкладка Raw

- Многострочное поле (аналогично Raw в Edit Outbound): JSON одного объекта правила маршрутизации sing-box (поля из [Route Rule](https://sing-box.sagernet.org/configuration/route/rule/)).
- При открытии диалога **добавления**: в Raw подставляется пример/заготовка (например `{ "domain": ["example.com"], "outbound": "proxy" }`) или пустой объект `{}` с комментарием.
- При открытии диалога **редактирования**: в Raw подставляется текущее правило в виде отформатированного JSON (из `editRule.Rule.Rule`).
- При переключении **Form → Raw**: в поле Raw подставляется JSON, соответствующий текущему состоянию формы (собранный через ту же логику, что и при Save для Form).
- При переключении **Raw → Form**: опционально — попытка распарсить JSON и выставить тип/поля формы; при неудаче или неоднозначности оставить форму как есть или переключить тип на «Custom JSON» и подставить сырой JSON в поле Custom. Минимально допустимо: при Raw → Form не менять форму, а при следующем Save использовать данные формы (т.е. Raw теряется при переключении на Form без сохранения). Конкретное поведение — в PLAN.
- Валидация при Save с активной вкладкой Raw: JSON должен быть валидным объектом; желательно проверять наличие `outbound` или `action`. При ошибке — сообщение пользователю, не закрывать диалог.
- Подсказка или кнопка **Documentation**: ссылка на документацию sing-box Route Rule (как в Edit Outbound — кнопка Documentation для Raw).

### 2.3 Rule name и Outbound при Raw

- **Rule name (label):** остаётся общим для обеих вкладок (вводится на форме); при сохранении из Raw label берётся из поля Rule name на форме.
- **Outbound:** либо входит в JSON на Raw (поле `outbound` в объекте правила), либо при сохранении из Raw подставляется из выбора Outbound на форме, если в JSON нет `outbound`/`action`. Требование: при Save с вкладки Raw итоговый объект правила должен содержать outbound/action; источник — JSON или форма.

### 2.4 Критерии приёмки

1. В диалоге Add Rule / Edit Rule есть две вкладки: Form и Raw.
2. На вкладке Raw — многострочное поле с JSON объекта route rule; при открытии (новое правило) — заготовка/пример, при редактировании — текущее правило в JSON.
3. При переключении Form → Raw в поле подставляется JSON, собранный из текущей формы.
4. При Save с активной вкладкой Raw правило сохраняется из распарсенного JSON; валидация (валидный JSON, наличие outbound/action); при ошибке — сообщение, диалог не закрывается.
5. Rule name всегда из формы; outbound при сохранении из Raw — из JSON или подстановка из формы, если в JSON нет.
6. Есть ссылка/кнопка на документацию sing-box Route Rule.

## 3. Ограничения

- Язык UI (подписи вкладок, кнопки, сообщения) — английский (CONSTITUTION).
- Синхронизация Raw ↔ Form при переключении вкладок может быть односторонней (Form → Raw при переключении на Raw); обратная синхронизация Raw → Form опциональна или упрощена (см. PLAN).
- Правило, сохранённое из вкладки Raw, не привязывается к типам формы (IP/Domain/Process/Custom JSON): это всегда произвольный JSON объекта route rule, без требования соответствия нашему шаблону.
