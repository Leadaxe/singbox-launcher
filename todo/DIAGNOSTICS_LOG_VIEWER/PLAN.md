# План: Отдельное окно логов (архитектура для согласования)

## 1. Общая идея

- На вкладке **Diagnostics** — кнопка «Open logs» (или «Logs»); по нажатию открывается **отдельное окно** (fyne.Window). В окне — **две вкладки**: **Internal** и **Core**.
- **Вкладка Internal**: источник — только перехват в `debuglog.Log()` (sink). На вкладке задаётся **уровень** (Error, Warn, Info, Verbose/Debug, Trace); в список попадают только события с уровнем не ниже выбранного. При открытии окна регистрируется sink через `debuglog.SetInternalLogSink(fn)`, при закрытии — `debuglog.ClearInternalLogSink()`. Sink не должен блокировать (push в channel, обновление UI через fyne.Async). Файл для Internal не используется.
- **Вкладка Core**: источник — только файл `logs/sing-box.log` (tail при открытии вкладки и по «Refresh»). Фильтр по уровню не применяется; уровни в строках можно отображать визуально (парсинг).

## 2. Компоненты

| Компонент | Назначение |
|-----------|------------|
| **Чтение логов (Core)** | Tail по файлу `logs/sing-box.log`: прочитать последние N строк (макс. 300 в отображении). Для Internal файл не читается; в виджете Internal хранить/показывать не более 300 последних событий. |
| **Парсинг уровня** | Internal: уровень приходит в sink из `debuglog.Log()`. Core: по ключевым словам в строке (error, warn, info, debug, trace) для визуальной подсветки; фильтр по уровню для Core не применяется. |
| **Перехват internal** | В `internal/debuglog`: при каждом `Log()` после вывода в log вызывать зарегистрированный sink `func(level Level, line string)` если он установлен. Публичный API: `SetInternalLogSink(fn)`, `ClearInternalLogSink()`. Sink вызывается из любой горутины; окно по выбранному уровню показывает только подходящие записи Internal (level не ниже выбранного). Sink не должен блокировать. |
| **Фильтр по уровню** | Только на вкладке Internal. Переключатель уровня: Error, Warn, Info, Verbose (Debug), Trace. В список попадают только события с уровнем не ниже выбранного. На вкладке Core фильтр по уровню не используется. |
| **UI** | Fyne: кнопка на вкладке Diagnostics открывает новое окно (fyne.Window). В окне — **две вкладки** (TabItem): **Internal** и **Core**. На вкладке Internal: выбор уровня, read-only виджет с логами из sink, подсветка уровня. На вкладке Core: кнопка «Refresh», read-only виджет с логами из файла (tail), подсветка уровня по парсингу. При открытии окна — регистрация sink в debuglog; при закрытии — снятие sink. Окно закрывается по кнопке или крестику. |

## 3. Поток данных

1. Пользователь на вкладке Diagnostics нажимает «Open logs» → открывается отдельное окно с двумя вкладками: **Internal** и **Core**. При открытии окна вызывается `debuglog.SetInternalLogSink(callback)` (sink активен пока окно открыто).
2. **Вкладка Internal**: выбор уровня (Error / Warn / … / Trace); перехват в `Log()` передаёт в callback `(level, line)`; на вкладке отображаются только строки с уровнем не ниже выбранного. Визуальная индикация уровня.
3. **Вкладка Core**: при переключении на неё или по «Refresh» загружается tail из файла `logs/sing-box.log`; все строки показываются, фильтр по уровню не применяется. Визуальная индикация уровня по парсингу строки.
4. При закрытии окна вызывается `debuglog.ClearInternalLogSink()`.

## 4. Размещение кода

- **Tail-чтение (только для Core)**: функция в `internal/` или `core/services/file_service.go`: `ReadLastLines(path string, maxLines, maxBytes int) ([]string, error)` для файла `logs/sing-box.log`. Internal из файла не читается.
- **Парсинг уровня**: для Core — по ключевым словам в строке (для подсветки); для Internal уровень приходит из sink.
- **debuglog**: экспорт `SetInternalLogSink(fn func(Level, string))` и `ClearInternalLogSink()`. В `Log()` после записи в log вызывать sink под RLock, если установлен. Sink предназначен для окна логов; callback не должен блокировать.
- **UI**: в `ui/diagnostics_tab.go` — кнопка «Open logs»; по нажатию создаётся отдельное окно (логи можно вынести в `ui/log_viewer_window.go`). В окне — **две вкладки** (Fyne TabContainer): **Internal** (выбор уровня + виджет логов из sink) и **Core** (кнопка Refresh + виджет логов из файла). При показе окна — SetInternalLogSink, при закрытии — ClearInternalLogSink. Путь к файлу Core — через `AppController.FileService.ExecDir` и константы.

## 5. Решения для согласования

| Вопрос | Предложение | Альтернативы |
|--------|-------------|--------------|
| **Обновление** | Только по кнопке «Refresh». | Добавить опциональное автообновление по таймеру при открытом окне логов (таймер отменять при закрытии окна). |
| **Структура окна** | Две вкладки: Internal и Core; на Internal — уровень и логи из sink, на Core — Refresh и логи из файла. | — |
| **Лимит** | Макс. 300 строк (событий) в отображении на каждой вкладке (Internal и Core). При tail для Core читать из файла не более 300 последних строк. | Меньше/больше по желанию. |
| **Путь к файлу Core** | Брать из `FileService`: `filepath.Join(ac.FileService.ExecDir, "logs", constants.ChildLogFileName)` для sing-box.log. Internal из файла не читается. | — |

## 6. Риски и ограничения

- Файлы логов могут быть открыты на запись (MainLogFile, ChildLogFile); чтение на Windows/других ОС обычно возможно одновременно с записью. Если окажется иначе — учесть копирование в буфер или предупреждение.
- Большие файлы: не читать целиком, только tail (seek с конца, читать блоками до набора N строк или N байт).
