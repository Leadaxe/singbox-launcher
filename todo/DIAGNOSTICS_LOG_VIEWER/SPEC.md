# Спецификация: Окно логов (Diagnostics)

## 1. Проблема

На вкладке **Diagnostics** сейчас есть только инструменты проверки IP (STUN, ссылки на 2ip, Yandex и т.д.). Пользователь не может посмотреть логи приложения и ядра (sing-box) без открытия файлов вручную.

## 2. Цель

Добавить **отдельное окно просмотра логов**, открываемое с вкладки Diagnostics (кнопка «Open logs» / «Logs»). В окне — **две вкладки**: **Internal** и **Core**.

- **Вкладка Internal**: логи лаунчера из перехвата `debuglog.Log()` (sink). На вкладке задаётся **уровень** (Error | Warn | Info | Verbose/Debug | Trace); в список попадают только события с уровнем не ниже выбранного (перехват передаёт каждую запись в sink `(level, line)`, окно отображает только подходящие).
- **Вкладка Core**: логи sing-box из файла `logs/sing-box.log` (tail при открытии вкладки и по «Refresh»). Фильтр по уровню не применяется; уровни в строках можно отображать визуально (парсинг).

## 3. Требования (кратко)

- На вкладке Diagnostics — кнопка, по нажатию открывается **отдельное окно** с просмотром логов (read-only).
- В окне логов — **две вкладки**:
  - **Internal**: выбор **уровня** — Error | Warn | Info | Verbose (Debug) | Trace (как в `debuglog.Level`). В список дублируются только события, попадающие под выбранный уровень (перехват в `debuglog.Log()` → sink `(level, line)`; окно решает, показывать ли строку). Визуальная индикация уровня каждой записи (цвет/иконка/метка).
  - **Core**: содержимое из файла `logs/sing-box.log` (tail), кнопка «Refresh», визуальная индикация уровня по парсингу строки; фильтр по уровню не используется.
- Источники данных:
  - **Internal**: только перехват в `debuglog.Log()` (sink), в реальном времени при открытом окне. Формат строк — префиксы `[DEBUG]`, `[INFO]`, `[WARN]`, `[ERROR]`; уровень передаётся в sink.
  - **Core**: только файл `logs/sing-box.log`; объём — последние N строк или N КБ (tail), обновление по «Refresh» и/или по таймеру (опционально).
- Тексты в UI — только английский (constitution).

## 4. Критерии приёмки

- Пользователь открывает окно логов с вкладки Diagnostics. На вкладке **Internal** задаёт уровень (Error / Warn / Info / Verbose / Trace) — отображаются только события, проходящие фильтр; перехват в `debuglog.Log()` дублирует подходящие записи. На вкладке **Core** показывается содержимое файла без фильтра по уровню.
- Уровень критичности каждой записи виден (цвет, иконка или подпись).
- При отсутствии файла лога или ошибке чтения — корректное сообщение (например, «Log file not available»), без падения приложения.
- Сборка и тесты проходят; логирование через debuglog в новых путях кода.

## 5. Вопросы для согласования (архитектура)

Ниже в **PLAN.md** предложен вариант архитектуры. Нужно согласовать:
- Чтение логов: только из файлов (tail) или ещё буфер в памяти для «текущей сессии»?
- Обновление: только по кнопке Refresh или + автообновление по таймеру при открытом окне логов?
- Ограничение: до 300 строк (событий) в отображении на каждой вкладке (согласовано в PLAN).

После согласования архитектуры будет развёрнут **PLAN.md** и **TASKS.md**.
