# What's New from 0.8.8.1

## Version 0.8.8.2 Release Notes — Hotfix follow-up

Cleanup of the v0.8.8.1 hotfix. The `@varname` substituter shipped with two embarrassing bugs that nobody hit (because the hard-coded fallback table covered them), but they meant user overrides from the Settings tab were silently ignored. Plus a broader path-invariant refactor so this kind of thing can't happen again.

---

## EN

### Fixed

- **`@varname` substitution now reads the right `state.json`.** v0.8.8.1's `BuildVarSubstituterFromDisk` looked at `bin/state.json`; the actual path is `bin/wizard_states/state.json`. And the `settings_vars` block was decoded as a `map[string]string` when on disk it's a `[{name, value}, ...]` array. Both bugs combined silently dropped user overrides from the Settings tab — `urltest_interval = 1m` set in the UI never made it into the generated config. The hard-coded URLTest fallback table covered the visible breakage in v0.8.8.1, so nobody hit a regression versus v0.8.8 (which was completely broken), but anyone who customized URLTest values just got the template default. Now both reads use the canonical paths and the documented schema.

### Changed — Path invariants

To prevent the above (a casual reader had to grep three packages to find where `state.json` actually lives), the canonical paths are consolidated:

- `internal/constants/constants.go` gains `WizardTemplateFileName`, `WizardStateFileName`, `WizardStatesDirName`. Previously these lived in `ui/wizard/template/loader.go`, `ui/wizard/models/wizard_state_file.go`, and `ui/wizard/business/state_store.go` respectively.
- `internal/platform/platform_common.go` gains three helpers — `GetWizardTemplatePath(execDir)`, `GetWizardStatesDir(execDir)`, `GetWizardStatePath(execDir)`. Code that needs to locate these files MUST go through them; no more hand-rolling `filepath.Join(execDir, "bin", "wizard_states", "state.json")` per call site.
- Existing constants in the `ui/wizard/...` packages are kept as re-exports of the canonical values (`StateFileName = constants.WizardStateFileName`, etc.) so external callers don't break.
- All existing call sites migrated to the helpers (`ui/wizard/template/loader.go`, `ui/wizard/business/state_store.go`, `core/config/varsubst.go`).

Tests pin the helpers (`platform_common_test.go`) and check the wizard state file via the real on-disk schema (`varsubst_test.go::TestBuildVarSubstituterFromDisk_StateOverrideRealSchema`).

### Technical / Internal

- `core/config/varsubst.go::BuildVarSubstituterFromDisk` signature changed from `(binDir string)` to `(execDir string)`. Internal — only `core/config_service.go` calls it.
- New `settingVarEntry` struct in `varsubst.go` mirrors `ui/wizard/models.SettingVar` JSON layout (kept local to avoid an upward import from `core/config/` to the ui-side package).

## RU

### Исправлено

- **`@varname` подстановка теперь читает правильный `state.json`.** В v0.8.8.1 `BuildVarSubstituterFromDisk` искал файл по пути `bin/state.json`; реальный путь — `bin/wizard_states/state.json`. Плюс блок `settings_vars` декодировался как `map[string]string`, а на диске лежит массив `[{name, value}, ...]`. Из-за двух багов сразу пользовательские override-ы из вкладки Settings молча игнорировались — `urltest_interval = 1m`, выставленный в UI, не доезжал до сгенерированного конфига. Hardcoded fallback-таблица URLTest в v0.8.8.1 покрывала видимую поломку, так что регрессии относительно v0.8.8 (где было сломано полностью) ни у кого не было, но любой кастомизатор URLTest получал template-default. Теперь оба чтения идут по каноническим путям и по документированной схеме.

### Изменено — Инварианты путей

Чтобы такое не повторялось (читателю приходилось grep'ать три пакета, чтобы понять, где реально лежит `state.json`), канонические пути сведены:

- `internal/constants/constants.go` получил `WizardTemplateFileName`, `WizardStateFileName`, `WizardStatesDirName`. Раньше они лежали в `ui/wizard/template/loader.go`, `ui/wizard/models/wizard_state_file.go` и `ui/wizard/business/state_store.go` соответственно.
- `internal/platform/platform_common.go` получил три helper-а — `GetWizardTemplatePath(execDir)`, `GetWizardStatesDir(execDir)`, `GetWizardStatePath(execDir)`. Код, которому нужны эти файлы, ОБЯЗАН ходить через них; никаких больше самосборок `filepath.Join(execDir, "bin", "wizard_states", "state.json")` на каждом callsite.
- Существующие константы в `ui/wizard/...` пакетах сохранены как re-exports канонических значений (`StateFileName = constants.WizardStateFileName` и т.д.), чтобы внешние callers не сломались.
- Все существующие callsite-ы мигрированы на хелперы (`ui/wizard/template/loader.go`, `ui/wizard/business/state_store.go`, `core/config/varsubst.go`).

Тесты пинят хелперы (`platform_common_test.go`) и проверяют чтение wizard state по реальной on-disk схеме (`varsubst_test.go::TestBuildVarSubstituterFromDisk_StateOverrideRealSchema`).

### Техническое / Внутреннее

- Сигнатура `core/config/varsubst.go::BuildVarSubstituterFromDisk` изменена с `(binDir string)` на `(execDir string)`. Внутренняя — вызывает только `core/config_service.go`.
- Новая структура `settingVarEntry` в `varsubst.go` зеркалит `ui/wizard/models.SettingVar` JSON-layout (локально, чтобы не тянуть импорт из core/config/ в ui-слой).
